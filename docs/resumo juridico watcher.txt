# PROJETO JURIDICO-WATCHER – RESUMO COMPLETO

## 1. VISÃO GERAL
Programa desktop leve, multiplataforma (Windows + Linux), escrito em Go, que monitora pastas do sistema de arquivos e processa automaticamente documentos PDF que possuam camada de texto. O processamento é baseado em regras configuráveis pelo usuário: cada regra define quais pastas observar, quais variáveis extrair do texto do documento, sob quais condições executar, e quais ações tomar (criar pasta, mover, renomear) usando as variáveis extraídas.

O cenário de uso principal é jurídico/notarial: documentos como procurações são digitalizados para PDF, o programa identifica automaticamente (ex: nome do outorgante) e organiza os arquivos em pastas com nomes padronizados.

## 2. STACK TECNOLÓGICA (DECISÕES TOMADAS)
- **Linguagem:** Go (módulo: github.com/F3rreir4L19/juridico-watcher, binário: juridico-watcher)
- **GUI:** Fyne (fyne.io/fyne/v2) – nativa, leve, binário único, sem deps externas em runtime, OpenGL, Multiplataforma.
- **Banco de dados:** SQLite via modernc.org/sqlite (driver Go puro, sem CGO, cross-compile simples)
- **Watcher de filesystem:** github.com/fsnotify/fsnotify (padrão Go, suporta inotify e ReadDirectoryChangesW)
- **Extração de texto de PDF:** github.com/ledongthuc/pdf (Go puro, para PDFs com camada de texto; sem dependência de OCR)
- **Geração de PDFs de teste:** github.com/jung-kurt/gofpdf
- **Testes:** stdlib testing + github.com/stretchr/testify (asserts)
- **Migrations:** Sistema interno via embed.FS (sem dependência externa)
- **Logs:** log/slog (stdlib Go 1.21+)

## 3. DECISÕES DE DESIGN (TODAS CONFIRMADAS)
- **Aplicabilidade vs Match separados:** A regra roda extração em todo PDF das pastas listadas; as ações só executam se as condições baterem. Sem condições = executa sempre após extrair.
- **Múltiplas regras:** Executam todas que derem match, na ordem de prioridade. Se uma regra moveu o arquivo, as seguintes são puladas (caminho original não existe mais). Campo "priority" na regra (menor = primeiro).
- **Política de colisão de nome:** Renomear com sufixo numérico: "arquivo (2).pdf", "arquivo (3).pdf"...
- **Estabilização de arquivo:** Ao detectar arquivo novo (watch contínuo), aguarda tamanho parar de mudar por 2 ciclos de 500ms antes de processar. Timeout máximo de 30s.
- **PDF sem texto extraível:** Marca como "no_text", não tenta OCR.
- **Delimitadores de extração:** Literais, case-insensitive, primeira ocorrência, sem regex no v1. Delimitador vazio = início/fim do texto.
- **Condições:** Operadores "eq", "neq", "contains", "not_contains". Combinação por AND (sem OR no v1).
- **Deleção de pasta monitorada:** Bloqueada se há regras que a referenciam. Usuário deve editar/deletar as regras primeiro. Garantido via FK ON DELETE RESTRICT + verificação no repo.
- **Watch recursivo por padrão:** Monitora subpastas, com opção de desativar (checkbox).
- **Sem system tray no v1:** Apenas janela aberta. System tray fica para v2.
- **Interpolação de variável inexistente:** Retorna string vazia + loga warning (não deixa o placeholder literal no nome do arquivo/pasta).

## 5. MODELO DE DADOS (ENTIDADES PRINCIPAIS)

**Watch:** id, name (único), path (absoluto), active (bool), recursive (bool), created_at, updated_at.

**Rule:** id, name (único), priority (int), active (bool).
- Relacionamentos N:N com Watch via tabela rule_watches.
- Contém slices: Extractions[], Conditions[], Actions[] (cada uma com order e referência rule_id).

**Extraction:** id, rule_id, variable_name, start_delim, end_delim, order.

**Condition:** id, rule_id, variable_name, operator (eq/neq/contains/not_contains), value, order.

**Action:** id, rule_id, type (create_folder/move/rename), target (string com interpolação {var}), order.

**ProcessedDoc:** id, file_hash (SHA-256 do conteúdo), original_path, rule_id, status (success/no_match/failed/skipped_moved/no_text), error_msg, processed_at. UNIQUE(file_hash, rule_id).

## 6. FLUXO DO MOTOR DE REGRAS (PIPELINE)
1. Detecta arquivo PDF (scan inicial, watch contínuo ou botão "atualizar").
2. Estabiliza tamanho (só no watch).
3. Calcula SHA-256. Se já processado por todas as regras aplicáveis, pula.
4. Extrai texto do PDF.
5. Lista regras ativas aplicáveis à pasta, ordenadas por prioridade.
6. Para cada regra:
   a. Aplica extrações (delimitadores → mapa de variáveis).
   b. Avalia condições (todas com AND). Se false, registra "no_match".
   c. Se true, executa ações em sequência, interpolando {var} em cada target.
   d. Registra resultado em processed_documents.
   e. Se uma ação moveu o arquivo, regras seguintes são puladas (skipped_moved).
